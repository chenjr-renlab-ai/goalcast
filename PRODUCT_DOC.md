# 预言者议会 · 产品与技术文档

> Oracle Council — AI Multi-Agent Football Prediction System

---

## 一、产品定位

**核心命题：把 AI 预测的推理过程本身做成内容。**

传统预测平台给用户一个数字，用户不知道为什么。预言者议会反过来：让六位拥有不同专业背景的 AI 议员公开辩论，用户全程旁观，最终看谁说服了谁——预测结论从辩论中自然浮现，而非从黑箱中输出。

**差异点**：用户消费的不是结论，是一场有戏剧张力的推理过程。

---

## 二、核心亮点

### 1. 六角色分工议会体制
六位 AI 议员各持不同信息视角，无法看到彼此的数据：

| 议员 | 角色设定 | 专属数据权限 | Temperature |
|------|---------|-------------|-------------|
| 数据帝·冰狗 | B站数据UP主 | 近期战绩、积分数据 | 0.3 |
| 月影姐 | 玄学预测博主 | 日期/地点/数字玄学 | 1.1 |
| 老球迷 | 60年球龄历史学家 | H2H历史、赛事背景 | 0.5 |
| 赌狗本狗 | 职业盘口分析师 | 赔率走势、EV数据 | 0.7 |
| 碎碎念 | 球员心理观察员 | 新闻、士气状态 | 0.6 |
| 议长 | 直播间主播 | 全局数据（促进辩论） | 0.25 |

**信息不对等是系统最核心的设计**：每个议员只能看到自己权限内的数据，必须在辩论中向其他议员"披露"自己的依据，创造真实的信息差和惊喜时刻。

### 2. 动态辩论机制（区别于"平行独白"）

- **反应机制**：每位议员发言后，与其立场最相悖的议员自动触发简短反应（15-25字），形成真实对话
- **矛盾制造者**：主持人识别最大分歧并点名交锋，话术从"总结"升级为"X认为...，Y认为...——X，你如何回应？"
- **立场动态追踪**：每次发言实时更新 agentStances，立场改变自动记录 pivotMoment

### 3. 结构化黑板协同（Blackboard Pattern）

```
blackboard = {
  facts:        已确认事实（可引用）
  claims:       争议主张（附来源）
  disputes:     活跃分歧列表（自动检测）
  agentStances: 各议员实时立场 {pick, conf}
  consensusLevel: 共识度 0-1（基于标准差计算）
  pivotMoments: 立场变化历史
  monitorLog:   实时监控日志流
}
```

共识度计算：`1 - min(std_dev(homeProbs) × 2, 1)`，高共识可触发提前收局。

### 4. 概率校正层（W-5 Rebalancer）

参考 WINNER12/W-5 框架的核心发现，纯 LLM 系统性低估平局和冷门概率：

```
修正规则：
- 平局修正：draw < 22% 时，提升至 max(draw,22) × 1.15
- 冷门修正：赔率差 > 2.0 时，冷门方 +7.5%
- 最大调整幅度：单项不超过 ±12%
- 自动重新归一化

EV计算（参考 Footixify）：
evHome = (home/100) × homeOdds - 1
正EV = 市场低估，存在超值
```

### 5. 恶魔代言人机制（信息不对等版）

每场议会随机指派一名议员为恶魔代言人，但不是强制反对，而是只给他看对主队不利的数据——让他基于真实存在的反向证据自然形成对立立场。投票时恢复完整信息，议会结束后揭示。

> 设计来源：ICLR 2025 MAD 研究发现强制反对派设计会降低预测质量；信息不对等版本保留了戏剧性而不引入虚假论点。

### 6. 用户对赌系统

开局前用户亮出自己的预测牌，结束后与议会结论对比，胜率跨场景累计。把被动观看变成有代入感的竞技。

### 7. LLM 叙事种子预生成（路径 B）

用户点击比赛后，立即后台触发：
1. 拉取真实英超数据（football-data.org + the-odds-api）
2. 调用 LLM 为5位议员各生成50-80字叙事种子（基于真实战绩/赔率/积分）
3. 种子就绪后议会才能开始

每个议员基于真实数据发言，而非凭空虚构。

### 8. 实时监控面板（/monitor.html）

后台可视化面板，通过 SSE 实时推送：
- 各议员立场条形图（含置信度）
- 共识度大数字仪表
- 活跃分歧列表
- 共享记忆流（agent 发言耗时、立场变化、分歧检测实时滚动）

## 四、数据流

```
用户打开页面
  → server 启动时后台预热：拉取英超赛程 + 积分榜
  → 前2场比赛自动预生成叙事种子

用户选择比赛
  → POST /api/match/:id/prepare（非阻塞）
  → 后台：拉战绩 + H2H + 赔率 → LLM 生成5个叙事种子（约4-6秒）
  → 前端轮询 /api/match/:id/ready，就绪后按钮变绿

用户选择预测 → 点击召开议会
  → SSE /api/run 建立连接
  → Phase: opening → initial（每人发言+reaction） → debate（矛盾制造+3轮对线） → vote（终投+恶魔揭示）
  → 每轮 emit blackboard_update → 前端立场面板实时更新
  → summary：server 拦截，调用 rebalance() 校正概率，计算 EV，返回增强结果

用户看到结果
  → 议会概率（校正后）+ EV超值指示 + 恶魔代言人揭示
  → 与用户预测对比，更新跨场景胜率
  → 后台 saveEpisode() 写入长期记忆
```

## 五、借鉴来源与设计决策

### 借鉴自 WINNER12/W-5 框架
- **概率校正层（Rebalancer）**：W-5 框架用 Gemini 作为概率重平衡器，对平局(+10%)和冷门(+25%)修正。我们实现了规则化的校正层，不依赖额外 LLM 调用
- **核心洞察**：纯 LLM 对平局概率系统性低估，需要独立校正步骤

### 借鉴自 Footixify
- **EV（期望价值）显示**：不只展示"谁会赢"，而是展示议会概率 vs 赔率隐含概率的差值（正EV=超值）
- 赌狗议员的核心职责对应调整：从"解读赔率"到"发现市场定价偏差"

### 借鉴自 SportBot AI
- **实时新闻情绪**：心理学家议员的数据权限设计为"新闻/士气状态"，对应 SportBot AI 的社交情绪注入方案（当前基于 briefing 字段，可后续接入 gstack 爬取）

### 来自 ICLR 2025 MAD 研究的警示
- **放弃强制反对派**：研究发现 Multi-Persona（强制恶魔代言人）框架显著降低预测质量——被强制反对的 agent 会把正确答案带偏
- **我们的调整**：恶魔代言人改为"信息不对等版"——只给他看反向证据，让他基于真实存在的数据自然形成对立立场
- **另一个发现**：增加辩论轮次和 agent 数量不一定提升准确度。我们的应对：辩论目的是内容和可解释性，准确度通过独立的 Rebalancer 层保证

### 自主设计的部分
- **多温度配置**：根据角色特点差异化 temperature（mystic=1.1 创意感，stat=0.3 严谨，moderator=0.25 稳定）
- **反应机制**：立场最相悖的议员自动触发简短回应，把"平行独白"变为真实对话
- **实时监控面板**：面向开发者/分析者的可观察性界面

## 七、待完善方向（已规划）
