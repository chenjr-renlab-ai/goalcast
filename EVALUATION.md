## 二、内容质量指标（Content Quality）

### 2.1 发言差异度（Diversity Score）

**定义**：同一场议会中，5个expert的发言在立场、方法论、数据引用、结论上的差异程度。

**为什么重要**：差异度=戏剧张力。全押同一边=无聊，三方博弈=精彩。

#### 指标 D1：立场分布熵（Shannon Entropy）

```
H = -Σ p(outcome) × log2(p(outcome))

计算方法：统计5个expert的winner字段分布
- 例：5人全押home → H = 0（最低，极其无聊）
- 例：3押home、1平、1away → H = 1.37（有趣）
- 例：2押home、2平、1away → H = 1.52（理想）

基准：H ≥ 1.0 为合格，H ≥ 1.3 为优秀
当前实测（估算）：约 1.1-1.4
```

#### 指标 D2：方法论碰撞率（Clash Rate）

```
定义：辩论轮次中显式引用METHOD_CLASH要点的比例
计算：检测发言文本中是否包含"你的模型"/"你的盘口"等对方框架词
目标：≥ 70% 的辩论轮次有明确方法论交锋
当前：需实测，预估约60-70%
```

#### 指标 D3：终投重复率（Vote Repetition Rate）

```
定义：终投阶段5人发言中，重复说同一件事（同一论点、同一比赛细节）的比率
计算：人工标注 or 文本相似度检测（cosine similarity > 0.7 = 重复）
目标：重复率 < 15%（5人中最多0-1次出现相同核心论点）
历史问题：v4.7前约40-50%，v4.8 buildVoteDirective后预计 < 20%
```

### 2.2 数据引用真实性（Data Authenticity）

**定义**：发言中引用的数字是否来自真实API数据，而非LLM虚构。

#### 指标 A1：数据字段覆盖率（Data Field Coverage）

```
定义：enrichMatchBriefing()执行后，AGENT_DATA_ACCESS中的字段有多少非空
关键字段清单（共15个）：
  homeForm, awayForm, standings, xg_note, avgHomeGoals, avgAwayGoals
  odds.home, odds.draw, odds.away, oddsMove, ev
  h2h, historicalNote, h2hScoreFreq
  news（FPL或赛前新闻）

目标：≥ 10/15 字段有真实数据（不含"无"/"暂无"/"数据暂无"）
当前：约8-11/15（FPL可用时更高）
触发降级提示：_fplAvailable=false时UI显示橙色警告
```

#### 指标 A2：数字引用率（Numeric Citation Rate）

```
定义：发言文本中出现真实数字（百分比/赔率/场次/比分）的比例
方法：正则提取数字，与briefing字段对比，判断是否属于已知数据
目标：≥ 80% 的初判发言包含至少1个可追溯的真实数字
注：月影姐数据完全模拟，排除在计算范围之外
```

### 2.3 叙事张力（Narrative Tension）

**定义**：一场议会是否有明确的"剧情弧"——悬念、转折、高潮、结局。

#### 指标 T1：立场转向率（Pivot Rate）

```
定义：一场议会中发生立场转向（pivot）的次数
blackboard.pivotMoments 统计
目标：≥ 1次pivot/场 为有趣，≥ 2次为精彩
触发条件：当前为Stance变化时自动记录
当前实测：约0-1次/场（需持续观察）
```

#### 指标 T2：辩论提前收场率（Early Stop Rate）

```
定义：因共识≥75%或连续2轮无转向而提前结束的比例
分析：
  - 全部5轮 → 双方僵持，内容丰富
  - 共识提前结束 → 说明一方论点强势
  - 僵局提前结束 → 说明双方各说各话，需改进
目标：约50%走完5轮，30%共识提前，20%僵局提前
```

#### 指标 T3：金句密度（Catchphrase Density）

```
定义：每场议会产生的catchphrase中，含有具体球员名或数字的比例
目标：≥ 80% 的catchphrase含球员名或数字
意义：高质量金句 = 可分享 = 传播力
```

## 三、用户体验指标（User Experience）

### 3.1 首次体验漏斗（First-Use Funnel）

**定义**：衡量新用户从打开页面到完成第一次完整议会的转化率。

```
漏斗步骤（理想路径）：
[1] 打开页面                       → 100%
[2] 看完引导弹窗 or 跳过            → 目标 ≥ 80%
[3] 选一场比赛                     → 目标 ≥ 70%
[4] 完成押注（不跳过）              → 目标 ≥ 50%
[5] 看完初判（约40s）               → 目标 ≥ 40%
[6] 看完完整议会（约6-8min）        → 目标 ≥ 25%
[7] 查看结果页                     → 目标 ≥ 20%
[8] 点击分享/下一场                → 目标 ≥ 10%

关键漏洞点（已知）：
  步骤[5]：初判40s无进度感（已修复：councilProgress指示器）
  步骤[1→2]：引导弹窗不说明等待时间（部分改善）
```

#### 指标 F1：5秒留存率（5s Retention）

```
定义：用户打开页面后5秒内没有关闭的比例
当前风险：空闲状态是暗色3D+静止小人，首屏无吸引力
改进措施：上场金句轮播(N5)、能量光圈(F7)
目标：≥ 60%

#### 指标 F2：议会完成率（Council Completion Rate）

```
定义：点击"召开议会"后，看到最终summary的比例
当前风险：初判40s是主要流失节点
改进措施：进度指示器(N4)、speaking_start动画(F2)

定义：召开议会时用户填写比分预测（不跳过）的比例

#### 指标 S1：分享率（Share Rate）

```
定义：看完议会后生成/保存战报图片的比例

目标：≥ 15%（接触到弹窗→点击生成）

```

## 四、预测性能指标（Prediction Performance）

### 4.1 绝对准确率（Absolute Accuracy）

**定义**：议会加权汇总给出的胜平负结论是否与实际结果一致。

#### 指标 P1：结果准确率（Outcome Accuracy）

```
统计方式：
  - 每场比赛结束后通过 POST /api/result 录入实际比分
  - memory.mjs 自动计算各agent和议会整体命中率
  - 主页显示近N场命中率

计算公式：
  correct / total * 100%

各agent命中率（长期跟踪）：
  stat, gambler, history, psych, mystic
  目标：≥ 45%（英超三元预测基准约33%，AI需明显超越）

议会加权汇总准确率目标：
  英超联赛：≥ 50%（较随机猜测+17个百分点）
  世界杯：≥ 45%（爆冷更多，基准更低）
```

#### 指标 P2：各Agent相对排名（Agent Leaderboard）

```
长期观察哪个agent准确率最高，用于：
1. 用户决策参考（"赌狗这周猜准了4场"）
2. 动态权重调整（高命中率agent在weightedSummaryCalc中权重更高）
3. 产品叙事（"AI准确率PK"营销角度）

期待模式：stat/gambler（基于客观数据）> mystic（模拟数据）
```

### 4.2 概率校准度（Probability Calibration）

**定义**：议会输出的概率是否与实际发生概率一致。

#### 指标 P3：Brier Score

```
定义：预测概率的均方误差
BS = (1/N) × Σ(predicted_prob - actual_outcome)²
完美预测 = 0

三元预测的随机基准：
  随机猜测时每个outcome概率=1/3
  BS = (1/3-0)² + (1/3-1)² + (1/3-0)² ÷ 3 ≈ 0.222
  ← 所以"BS≤0.22"其实是"约等于随机"，不是"优于随机"

正确目标：
  优秀：BS < 0.18
  合格：0.18 ≤ BS < 0.22
  不及格：BS ≥ 0.22（不如随机猜测）

计算工具：
  Python: from sklearn.metrics import brier_score_loss
  或：在 memory.mjs 中自动统计（待实现）
```

#### 指标 P4：平局低估修正效果（Draw Correction）

```
背景：纯LLM系统性低估平局概率（约15-18%），W-5校正到22%
验证方法：
  - 统计rebalancer修正前后的平局预测比例
  - 对比实际英超平局率（约25-28%）
目标：修正后平局率落在22-26%（而非修正前的15-18%）
```

### 4.3 EV正向率（Expected Value Rate）

**定义**：议会判断的 EV > 0 场次中，最终结果符合预测的比例。

```
EV > 0 = 市场低估，议会认为存在"超值"机会
计算：在 evHome/evDraw/evAway 任一 > 0 时，对应结果的命中率
目标：EV>0场次命中率 ≥ 55%（若模型有效，超值机会应该更容易命中）
注意：样本量需≥30场才有统计意义
```

## 五、技术性能指标（Technical Performance）

### 5.1 响应速度（Response Speed）

#### 指标 T1：初判完成时间（Initial Phase Duration）

```
定义：从Phase.initial开始到5个expert全部发言完成的时间
测量：后台logMonitor记录agent_start和agent_done时间戳
当前实测：约35-50秒（DeepSeek 7-9s/call × 5 = 35-45s）
TTFT（首字节）：~1s（speaking_start事件）
目标：≤ 40s完成，且用户体感"有事在发生"（进度指示器覆盖）
```

#### 指标 T2：单次Agent调用延迟（Agent Call Latency）

```
当前：DeepSeek-v3.2 实测 7-9s
基准：文档标称 2.2s（实际平台负载差距大）
对比模型（待测）：doubao-seed-code 文档标称 3.1s
目标：≤ 5s/call（则5人串行共25s，体验可接受）
```

#### 指标 T3：议会总时长（Total Council Duration）

```
典型时长：
  Opening(~10s) + Initial(~40s) + Debate(~60s) + Vote(~40s) = ~2.5min
  含reading delay等待 = 约5-8min

目标用户场景：赛前30分钟看完，8min以内是可接受的
```

### 5.2 稳定性（Stability）

#### 指标 S1：议会完成率（Session Success Rate）

```
定义：点击召开议会后，顺利走到summary（非错误终止）的比例
已知问题（已修复）：
  - Moonshot error 42限速导致卡死（v4.8已切火山方舟）
  - 并发导致display loop阻塞（v4.8已改串行）
当前预期：≥ 90%（仍有<10%可能遇到API超时）
降级机制：单agent失败→占位消息"信号丢失"→继续推进
```

#### 指标 S2：Agent失败降级率（Fallback Rate）

```
定义：5个expert中至少1个因API失败而使用占位消息的比例
当前机制：失败→3s重试1次→仍失败→发"信号丢失"占位
目标：全成功率 ≥ 85%；占位率 ≤ 15%
```

### 5.3 数据鲜活度（Data Freshness）

#### 指标 D1：赛前数据补全率（Briefing Enrichment Rate）

```
定义：enrichMatchBriefing()执行后，AGENT_DATA_ACCESS关键字段中非空字段的比例
关键字段（15个）：homeForm, awayForm, standings, odds, h2h, news等
目标：≥ 10/15（FPL可用时）, ≥ 7/15（FPL不可用时）
```

#### 指标 D2：缓存命中率（Cache Hit Rate）

```
定义：API请求中直接从.cache/读取（未发网络请求）的比例
缓存TTL设置：odds=6h, fixtures=12h, form=6h, standings=12h
意义：缓存命中=更快响应+节省API配额
目标：首次冷启动后，同一天内 ≥ 70% 命中缓存
```

## 九、核心矛盾与权衡

### 9.1 娱乐性 vs 可信度

```
用户反馈："看乐子可以，但不会信"

娱乐性驱动力：
+ 6个AI角色的个性对撞
+ 方法论互怼的戏剧感
+ 恶魔代言人揭晓的悬念

可信度信号：
+ 真实数据来源标注（AGENT_METHOD_LABEL）
+ 历史命中率展示（准确率徽章）
+ 概率校正层（W-5 Rebalancer）
+ EV超值指示（当前EV>0是否真的更容易中？需验证）

当前平衡点：
娱乐性优先，可信度作为辅助信号
不追求"预测工具"定位，而是"你可以参考的AI辩论秀"
```

### 9.2 速度 vs 质量

```
用户容忍等待时间：约8-10分钟（完整议会）
当前初判等待：35-50秒（最大痛点）

解决方案优先级：
1. 感知改善（进度指示器、TTFT动画）← 已实现
2. 速度提升（评估doubao-seed-code 3.1s）← 待测试
3. 真正流式（speech逐字出现）← 中期规划
4. 减少内容（缩短speech字数）← 会损失质量，最后考虑
```

### 9.3 英超 vs 世界杯

```
英超：
+ 全年持续运营
+ 数据完整（FPL/H2H/赔率）
+ 用户群较小（英超球迷）
- 情感烈度中等

世界杯：
+ 流量窗口集中（2026-06-11至07-19，共39天）
+ 中国队参赛（历史级事件）
+ 情感烈度极高
- 数据缺口（FPL不适用，需SofaScore）
- 有限的时间窗口

策略：英超是实验室，世界杯是上市场
```
